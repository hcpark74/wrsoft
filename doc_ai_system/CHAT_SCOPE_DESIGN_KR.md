# 대화 범위 설계안

이 문서는 문서 기반 챗봇에서 `대화 범위`를 어떻게 제어할지 정리한 설계안입니다.

핵심 원칙은 다음과 같습니다.

- 대화 범위는 별도 챗봇 인스턴스를 만드는 개념이 아니라, `검색 대상 범위`를 줄이는 방식으로 제어
- 범위 제어는 `스토어 -> 카테고리 -> 선택 문서`의 3단계로 점진적으로 좁힘
- 실제 구현은 `fileSearchStoreNames`, `metadata_filter`, 선택 문서 기반 필터링 조합으로 처리

## 1. 범위 모델

### 1단계. 스토어 범위

- 가장 큰 범위 단위
- 어떤 지식 저장소를 검색 대상으로 쓸지 결정
- 예: `company-main`, `legal-kb`, `product-docs`

의미:

- 서로 다른 도메인의 지식을 물리적으로 분리
- 권한, 운영, 배포 단위를 나누기 쉬움
- 멀티테넌트나 프로젝트별 분리에 적합

사용 예:

- 관리자 콘솔에서 `기본 스토어` 선택
- 위젯별로 특정 스토어만 연결
- Playground에서 스토어별 응답 품질 비교

## 2단계. 카테고리 범위

- 스토어 내부 문서를 분류하는 운영 메타데이터 범위
- 예: `company`, `legal`, `tech`, `hr`
- 카테고리는 독립 대화방이 아니라 `검색 필터` 역할

의미:

- 같은 스토어 안에서도 질의 범위를 실무 단위로 좁힘
- 운영자가 특정 업무 영역만 대상으로 테스트 가능
- 사용자에게는 보이지 않거나, 단순 주제 선택 UX로 노출 가능

사용 예:

- `category="legal"`
- `category="tech"`
- 특정 위젯은 `company`와 `legal`만 허용

## 3단계. 선택 문서 범위

- 가장 좁은 범위 단위
- 사용자가 명시적으로 선택한 문서 집합만 검색 대상으로 사용
- 예: 문서 목록에서 2개 PDF를 체크하고 그 안에서만 질의

의미:

- 답변의 근거 범위를 매우 명확하게 제한
- 검토, 비교, QA, 계약서 확인 같은 작업에 유리
- 일반 검색보다 정확한 검증형 UX에 적합

사용 예:

- `이 3개 문서 기준으로만 답변해줘`
- `선택한 문서만 포함`
- 검색 결과에서 문서를 pin 해서 후속 질문에 유지

## 2. 3단계 범위 관계

```text
Store
└─ Category Filter
   └─ Selected Documents
```

정리하면:

- 스토어는 `어느 지식베이스를 쓸지`
- 카테고리는 `그 안에서 어떤 주제를 볼지`
- 선택 문서는 `정확히 어떤 파일만 볼지`

## 3. UX 관점 역할 분리

### 스토어

- 관리자 중심 설정값
- 일반 사용자에게는 보이지 않거나 서비스 단위로 고정
- 예: 회사 사이트 챗봇은 `company-main`만 사용

### 카테고리

- 운영 필터 또는 주제 선택값
- 관리자에게는 강한 제어 수단
- 사용자에게는 `질문 분야 선택` 정도로만 노출 가능

### 선택 문서

- 사용자 액션 기반 범위 제한
- 가장 직관적이고 설명 가능한 범위 제어 방식
- `현재 선택된 문서 기준`이라는 문구로 안내 가능

## 4. 권장 우선순위

### 기본 모드

- 스토어 고정
- 카테고리 필터 선택 가능
- 선택 문서 없음

가장 무난한 운영 방식입니다.

### 정밀 모드

- 스토어 고정
- 카테고리 선택 가능
- 선택 문서 on/off 가능

계약 검토, 보고서 비교, 규정 확인처럼 근거 통제가 중요한 경우에 적합합니다.

### 관리자 테스트 모드

- 스토어 선택 가능
- 카테고리 필터 선택 가능
- 선택 문서 지정 가능

Playground에서 품질 점검용으로 가장 적합합니다.

## 5. API/구현 매핑

## 스토어 범위

- 구현 방법: `fileSearchStoreNames` 지정
- 효과: 검색 대상 스토어를 제한

예시:

```json
{
  "fileSearch": {
    "fileSearchStoreNames": ["fileSearchStores/company-main"]
  }
}
```

## 카테고리 범위

- 구현 방법: `metadata_filter` 사용
- 효과: 카테고리 메타데이터가 일치하는 문서만 검색

예시:

```json
{
  "fileSearch": {
    "fileSearchStoreNames": ["fileSearchStores/company-main"],
    "metadata_filter": "category=\"legal\""
  }
}
```

## 선택 문서 범위

- 구현 방식은 제품 설계에 따라 달라질 수 있음
- 기본 아이디어는 `사용자가 선택한 문서 집합만 검색 대상으로 제한`하는 것
- 구현 후보:
  - 문서별 고유 메타데이터를 부여하고 `metadata_filter`로 제한
  - 별도 세션 상태에 선택 문서 목록을 저장
  - 선택 문서 전용 임시 컬렉션/세션 범위를 구성

즉, 스토어/카테고리보다 선택 문서 범위는 제품 레벨 설계가 더 필요합니다.

## 6. 권장 제품 설계

### Admin

- 스토어: 명시적으로 선택 가능
- 카테고리: 필터로 적극 사용
- 선택 문서: Playground에서만 지원

### Floating Chat Widget

- 스토어: 위젯 설정에서 고정
- 카테고리: 기본적으로 숨기거나 단순 주제 선택으로 노출
- 선택 문서: 일반 위젯에서는 비노출

### Whole-page Chat / Fixed Chatbox

- 스토어: 고정
- 카테고리: 선택 가능
- 선택 문서: 문서 목록이 함께 있을 때만 지원

## 7. 추천 UX 문구

사용자에게는 기술 용어보다 범위 설명 문구가 중요합니다.

- 스토어: `현재 지식베이스: 회사 문서`
- 카테고리: `질문 분야: 법률/규정`
- 선택 문서: `선택한 3개 문서 기준으로 답변 중`

피해야 할 표현:

- `카테고리와 대화 중`
- `legal 챗봇으로 전환`

추천 표현:

- `법률 문서만 대상으로 검색`
- `선택한 문서 기준으로 답변`

## 8. 단계별 도입안

### 1단계

- 스토어 고정
- 카테고리 필터 지원
- 관리자 Playground에서만 범위 표시

### 2단계

- 위젯 설정에서 기본 카테고리 정책 추가
- 사용자용 주제 선택 UX 도입

### 3단계

- 선택 문서 기반 질의 지원
- `현재 답변 범위` 배지와 해제 UX 제공

## 결론

대화 범위는 별도의 챗봇을 여러 개 만드는 문제가 아니라, `검색 범위를 얼마나 정교하게 제한할 것인가`의 문제입니다.

가장 현실적인 설계는 다음과 같습니다.

- 스토어 = 지식베이스 단위
- 카테고리 = 운영/주제 필터
- 선택 문서 = 정밀 검토용 범위 제한

이 3단계를 분리하면 관리자 UI와 사용자 챗봇 UI 모두 훨씬 명확하게 설계할 수 있습니다.

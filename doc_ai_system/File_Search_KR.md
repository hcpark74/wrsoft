## 개요

`File Search`는 Gemini API에서 제공하는 RAG(Retrieval Augmented Generation) 기능입니다.  
문서를 업로드하면 내용을 청크(chunk)로 나누고, 임베딩을 생성해 검색 가능한 저장소에 인덱싱합니다. 이후 사용자 질의가 들어오면 의미 기반 검색(semantic search)으로 관련 문서 조각을 찾아 모델 응답의 근거로 활용합니다.

즉, 단순히 모델에 긴 문서를 직접 넣는 방식이 아니라, 필요한 문맥만 검색해 응답 정확도를 높이는 구조입니다.

---

## 주요 목적

- 업로드한 문서를 기반으로 더 정확한 답변 생성
- 문서 근거가 있는 응답 제공
- 대용량 문서를 효율적으로 검색
- 사내 문서/정책/매뉴얼/기술문서 기반 Q&A 구현

---

## 핵심 구성요소

### 1. File Search store

문서 임베딩과 인덱스를 저장하는 검색 저장소입니다.  
실제 검색은 이 저장소를 대상으로 수행됩니다.

### 2. File

원본 파일 객체입니다.  
Files API를 통해 먼저 업로드한 뒤 File Search store로 가져올 수 있습니다.

### 3. FileSearch tool

`generateContent` 호출 시 연결하는 도구입니다.  
어떤 store를 검색할지 지정하면, 모델이 해당 저장소를 검색해 관련 문맥을 찾아 응답에 활용합니다.

---

## 동작 방식

File Search의 기본 흐름은 다음과 같습니다.

1. `File Search store` 생성
2. 파일 업로드 또는 기존 파일 import
3. 파일 자동 청킹
4. 청크별 임베딩 생성
5. 검색 인덱스 구성
6. 질의 시 관련 청크 검색
7. 검색 결과를 모델 컨텍스트로 사용해 응답 생성

---

## 파일 등록 방식

## 1) 직접 업로드: `uploadToFileSearchStore`

파일 업로드와 store 반입을 한 번에 처리하는 방식입니다.

### 장점

- 구현이 단순함
- 빠르게 시작 가능
- 별도 Files API 관리가 필요 없음

### 특징

- 내부적으로 임시 `File` 객체가 생성됨
- 원본 파일 객체는 48시간 후 삭제될 수 있음
- 하지만 store에 인덱싱된 데이터는 유지됨

---

## 2) 분리 업로드: `files.upload` 후 `importFile`

먼저 Files API에 원본 파일을 업로드하고, 이후 File Search store로 import하는 방식입니다.

### 장점

- 파일 업로드와 검색 인덱싱을 분리 가능
- 기존 파일 관리 플로우와 통합하기 쉬움
- 업로드/가져오기 단계를 명확히 제어 가능

---

## 청킹(Chunking)

문서를 store에 넣으면 자동으로 여러 청크로 분할됩니다.  
필요하면 `chunking_config`를 통해 청킹 전략을 조정할 수 있습니다.

### 주요 설정값

- `max_tokens_per_chunk`
  - 청크 하나에 들어갈 최대 토큰 수
- `max_overlap_tokens`
  - 이전 청크와 겹치게 포함할 토큰 수

### 의미

- 청크가 너무 크면 검색 정밀도가 낮아질 수 있음
- 청크가 너무 작으면 문맥이 끊길 수 있음
- overlap은 문맥 손실을 줄이는 데 도움됨

### 실무 팁

- 처음에는 기본값으로 시작
- 검색 품질이 낮으면 chunk 크기와 overlap 조정
- 문서 성격(매뉴얼/FAQ/정책/코드)에 따라 최적값이 달라질 수 있음

---

## 의미 기반 검색(Semantic Search)

File Search는 일반 키워드 검색이 아니라 의미 기반 검색을 사용합니다.

### 동작 원리

- 문서를 임베딩 벡터로 변환
- 사용자 질의도 임베딩 벡터로 변환
- 벡터 간 유사도를 계산해 관련 청크 검색

### 장점

- 질의 표현이 문서와 정확히 일치하지 않아도 검색 가능
- 유사 의미 문장을 더 잘 찾을 수 있음
- 자연어 질의응답에 적합

---

## 메타데이터(Metadata)

문서에 사용자 정의 메타데이터를 추가할 수 있습니다.

### 예시

- `author`
- `year`
- `category`
- `department`
- `product`
- `version`

### 활용 방식

- 문서 import 시 메타데이터 추가
- 질의 시 `metadata_filter`로 특정 문서만 검색

### 예시 활용

- 특정 부서 문서만 검색
- 특정 작성자 자료만 검색
- 특정 버전 문서만 검색
- 특정 제품군 문서만 검색

### 장점

- 한 store에 여러 문서가 섞여 있어도 검색 범위를 제어 가능
- 검색 정확도 향상
- 멀티테넌트/멀티도메인 환경에 유리

---

## 인용(Citations)

File Search를 사용하면 모델 응답에 출처 정보가 포함될 수 있습니다.

### 확인 위치

- `grounding_metadata`

### 장점

- 어떤 문서가 근거였는지 추적 가능
- 응답 검증 가능
- UI에서 출처 표시 가능
- 신뢰성 향상

---

## Structured Output 지원

Gemini 3 계열부터는 File Search와 구조화 출력(JSON schema)을 함께 사용할 수 있습니다.

### 의미

검색 기반 응답을 자유 텍스트가 아니라 정해진 JSON 구조로 받을 수 있습니다.

### 활용 예시

- 계약서에서 금액/통화 추출
- 정책 문서에서 규정 값 추출
- 문서 기반 정보 추출 자동화
- 정형 데이터 파이프라인 구성

---

## 저장 수명과 보존 정책

### 원본 파일

- Files API에 업로드된 원본 파일은 48시간 후 삭제될 수 있음

### File Search store 데이터

- store에 인덱싱된 데이터는 TTL 없이 유지됨
- 직접 삭제하기 전까지 보존됨
- 모델 지원 종료 시 제거될 수 있음

### 정리

장기적으로 유지되는 것은 원본 파일이 아니라 `File Search store`의 인덱스 데이터입니다.

---

## File Search store 관리

File Search store는 다음 작업을 지원합니다.

- 생성
- 목록 조회
- 단건 조회
- 삭제

### 운영 권장 방식

문서 유형별로 store를 분리하는 것이 좋습니다.

예:

- `prod-hr-docs`
- `prod-tech-docs`
- `prod-policy-docs`
- `dev-internal-guides`

### 주의

문서에 따르면 store 이름은 globally scoped입니다.  
따라서 서비스명, 환경명, 프로젝트명을 포함한 이름 전략을 사용하는 것이 안전합니다.

---

## 문서 단위 관리

store 내부 문서는 개별적으로 관리할 수 있습니다.

지원 기능:

- 문서 목록 조회
- 문서 상세 조회
- 문서 삭제

즉, 전체 store를 지우지 않고 특정 문서만 제거할 수 있습니다.

---

## 지원 파일 형식

File Search는 매우 다양한 파일 형식을 지원합니다.

### 대표 예시

- 문서형: PDF, Word, Excel, PPTX, OpenDocument
- 데이터형: JSON, XML, CSV, SQL
- 텍스트형: TXT, Markdown, HTML, YAML
- 코드형: JavaScript, TypeScript, Python, Java, Go, Rust, C/C++, Shell 등

즉, 일반 문서뿐 아니라 코드베이스나 설정 파일 검색에도 활용할 수 있습니다.

---

## 제한 사항

### 기능 제한

- Live API에서는 지원되지 않음
- 다른 일부 도구와 동시에 사용 불가
  - 예: Google Search grounding
  - 예: URL Context

### 용량 제한

- 파일당 최대 크기: `100 MB`

### 프로젝트 전체 store 용량 제한

- Free: `1 GB`
- Tier 1: `10 GB`
- Tier 2: `100 GB`
- Tier 3: `1 TB`

### 권장 사항

- store 크기는 `20 GB 이하` 유지 권장
- store가 너무 커지면 검색 지연과 운영 복잡도가 증가할 수 있음

### 참고

백엔드 저장 용량은 단순 원본 파일 크기가 아니라, 원본 + 임베딩 + 인덱스를 포함해 계산됩니다.  
문서에 따르면 일반적으로 입력 데이터 크기의 약 3배 수준이 될 수 있습니다.

---

## 과금 구조

### 비용 발생 항목

- 최초 인덱싱 시 임베딩 생성 비용 발생
- 검색된 문서가 모델 컨텍스트로 들어가면 일반 입력 토큰처럼 과금

### 무료 항목

- 파일 저장소 보관
- 질의 시 임베딩 생성

### 해석

- 저장 자체는 부담이 적음
- 하지만 검색 결과가 길수록 모델 입력 토큰 비용은 증가할 수 있음

---

## 장점

- 의미 기반 검색 지원
- 문서 근거가 있는 응답 생성 가능
- 인용(citation) 확인 가능
- 메타데이터 기반 필터링 가능
- 구조화 출력과 결합 가능
- 다양한 문서 및 코드 파일 지원

---

## 주의할 점

- 청킹 전략이 검색 품질에 큰 영향을 줌
- store 설계를 잘못하면 운영이 복잡해질 수 있음
- retrieved context가 길면 토큰 비용 증가
- 다른 도구와 동시 사용 제약이 있음
- store 용량은 실제 입력 파일보다 더 크게 계산될 수 있음

---

## 실무 적용 권장 사항

### 1. store 분리 전략 수립

문서 성격별로 store를 나누는 것이 좋습니다.

예:

- 인사 문서
- 정책 문서
- 기술 문서
- 고객 지원 문서

### 2. 메타데이터 체계 정의

초기부터 아래 같은 메타데이터 체계를 정해두는 것이 좋습니다.

- `source`
- `department`
- `owner`
- `version`
- `updated_at`
- `product`

### 3. 청킹 기본값으로 먼저 시작

초기에는 기본 청킹을 사용하고, 검색 품질을 본 뒤 조정하는 것이 효율적입니다.

### 4. 출처 표시 UI 고려

사용자 응답에 citation 또는 문서명을 표시하면 신뢰성이 올라갑니다.

### 5. 비용 모니터링

다음 항목을 함께 관리하는 것이 좋습니다.

- 인덱싱 비용
- 질의 시 입력 토큰 비용
- store 총 용량
- 문서별 검색 품질

---

## 결론

Gemini `File Search`는 문서를 검색 가능한 지식 저장소로 변환해, 모델 응답의 정확도와 근거성을 높이는 RAG 기능입니다.

핵심 운영 단위는 `File Search store`이며, 단순 파일 업로드보다 검색용 인덱스 설계가 중요합니다. 또한 메타데이터 필터링, citation, structured output까지 지원하므로 실무형 문서 QA 시스템이나 지식 검색 서비스 구축에 적합합니다.
